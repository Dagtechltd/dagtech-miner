DagTech BlockDAG Node v2.0.1
Corruption-Proof Release

A hardened QNG / Qitmeer-fork node tuned for the BlockDAG mainnet. Soft-recovery from PathDB shutdown breadcrumbs, tunable diff-layer cap, IPFS snapshot bootstrap, and a local-only dashboard. No telemetry, no phone-home, no analytics. Your stats stay on your box.

1Install (one line)

Run on Linux (amd64 / arm64) as the user that will own the node. The installer is idempotent, verifies its own SHA-256, pulls the IPFS chain snapshot to skip the multi-day resync, and brings up a local dashboard you can open in your browser.

curl -fsSL https://miner.dagtech.network/v201/install.sh | bash
i

Pre-flight checks the installer runs: 8 GB RAM, 200 GB free disk, outbound TCP 8130 reachable, Docker present. It will refuse and explain rather than fail half-installed.

!

v2.0.1 is a one-way trip. Once the node shuts down cleanly on v2.0.1, the on-disk state format is incompatible with v1. Snapshot your data dir before upgrading if you want a rollback path.

2Local dashboard (auto-opens at http://127.0.0.1:8881)

After install, the node serves a local-only dashboard on loopback. Health, peers, sync progress, current block, RAM/CPU, recent shares, wallet inflow vs outflow - all rendered server-side from your own node. Nothing is sent anywhere.

# Open the dashboard
xdg-open http://127.0.0.1:8881    # Linux
open     http://127.0.0.1:8881    # macOS
start    http://127.0.0.1:8881    # Windows

# Or just point your browser at:

Why loopback only. The dashboard binds to 127.0.0.1 by design - your operational telemetry is your business. If you want to reach it from your laptop, use an SSH tunnel: ssh -L 8881:127.0.0.1:8881 user@your-node. No accounts, no API keys, no remote endpoints.

5What makes this build unique to DagTech

This is not a rebadged upstream binary. The fixes below were derived from real production incidents on our UAE mining node, and the engineering trail is in our public commit history.

01

The "Illegal withdrawal" bug fixed

Every other operator was told to run --cleanup and wipe their chain to genesis. We traced it to the shutdown tracker in qng-src/core/shutdown/tracker.go and shipped --rescue for soft, non-destructive recovery.

02

PathDB diff layer cap raised + tunable

v1 hardcoded maxDiffLayers=128 for vanilla Ethereum. GHOSTDAG consensus needs more headroom. We default to 256 and expose --state.diff-layers-max for memory-aware tuning.

03

Sync-stuck self-heal (30s watchdog)

If the chain head doesn't advance for 30 seconds while peers are connected, an in-process watchdog unsticks the import pipeline without restarting the binary. Catches the cases where the node sat silent for hours.

04

DagTech signature in --version

The running binary prints a DagTech signature tag at startup. Anyone can verify at a glance that what's executing is our build - not an upstream copy with the same version string.

05

Local dashboard, zero phone-home

The dashboard at 127.0.0.1:8881 is server-rendered from your own node. No analytics, no telemetry, no DagTech endpoints called from the binary. Your operational data never leaves your box.

06

Reproducible, hash-verified builds

SHA256SUMS for all platforms. The build is deterministic - 10/10 reproducibility verified across rebuild. GPG signature lands in v2.1.0.

6New flags

--rescue

Non-destructive recovery. Clears the stale shutdown breadcrumb, rolls back to the last fully-committed block, and resumes. Use this instead of --cleanup for OOM/SIGKILL/power-loss recovery.

--state.diff-layers-max=N

Caps in-memory PathDB diff layers. Default 256. Lower (128) for 8 GB RAM, raise (512+) on 32 GB+ pool backends. Each layer is ~30-60 MB of working set.

--cache=N

EVM cache in MB. Sensible: 2048 for 8 GB hosts, 4096 default for 16 GB, 8192 for 32 GB+.

--rescue --dry-run

Reports what --rescue would do without writing. Useful before a real rescue if you want to see which block it will roll back to.

7What v2.0.1 fixes

  • "Illegal withdrawal at block X" is now recognised as a shutdown breadcrumb (not a withdrawal). --rescue soft-recovers without wiping the chain.
  • "missing trie node" after unclean shutdown is auto-recovered on next start instead of refusing to boot.
  • maxDiffLayers is no longer hardcoded. Adapts to DAG reorg depth via --state.diff-layers-max.
  • PathDB flush / snapshot / reorg transient errors are distinguished from corruption. The node retries rather than panicking.
  • lockfile orphan on container kill no longer prevents restart; replaced by atomic PID-tracked lock.
  • Sync-stuck watchdog (30s) unsticks the import pipeline when chain head sits still with peers connected.
  • Local-only dashboard on 127.0.0.1:8881 - no remote stats endpoint.
  • Reproducible builds: SHA-256 signed, deterministic. 10/10 reproducibility verified.

8Chain snapshot (skip the 2.7-day resync)

A full resync from genesis takes ~2.7 days. The installer pulls Jeremy's auto-updating IPFS chain snapshot - SHA-verified, peer-list-bundled, ~10 GB compressed, refreshed continuously.

# The installer does this for you. Manual path:
SNAPSHOT_CID=$(curl -s https://miner.dagtech.network/v201/snapshot-cid.txt)
curl -L "https://${SNAPSHOT_CID}.ipfs.dweb.link/" -o snapshot.tar.zst
curl -L "https://${SNAPSHOT_CID}.ipfs.dweb.link/sha256" -o snapshot.tar.zst.sha256
sha256sum -c snapshot.tar.zst.sha256

# Then stop node, extract, restart:
docker stop blockdag-node
tar --use-compress-program=unzstd -xf snapshot.tar.zst -C /var/lib/blockdag/
docker start blockdag-node
i

Use the subdomain gateway form (CID.ipfs.dweb.link/) - it works. Path-style gateways return 404 for our snapshots, don't bother.

12How to sync to network tip (from scratch)

If you're starting cold or your chain head fell behind, here's the path with the lowest time-to-tip:

  1. Stop the node if it's running: docker stop blockdag-node
  2. Pull the IPFS snapshot - ~10 GB compressed, gets you to roughly head-minus-15-minutes.
  3. Verify the snapshot SHA against the published hash before extracting (the installer does this automatically).
  4. Extract into your data dir - tar --use-compress-program=unzstd -xf snapshot.tar.zst -C /var/lib/blockdag/
  5. Start the node - docker start blockdag-node
  6. Watch convergence at http://127.0.0.1:8881. Expect chain head to advance from the snapshot block to network tip in 5-15 minutes once peers are above 4.
  7. Verify you're at tip: dashboard shows synced: true and chain head equals network head from any public RPC.

If you'd rather sync from peers and skip the snapshot entirely: just start the node with an empty data dir. Expect ~2.7 days on a good connection, longer on a marginal one. Most operators don't want this.

13FAQ - operator answers

How do I restart my node? (per OS)

Docker (most operators):

docker restart blockdag-node for a soft restart. docker compose up -d --force-recreate blockdag-node if you've edited docker-compose.yml and need the changes picked up.

Linux systemd:

sudo systemctl restart dagtech-node + sudo journalctl -fu dagtech-node to tail logs.

Windows:

Restart-Service DagTechNode from an elevated PowerShell. Logs live at %ProgramData%\DagTech\node\logs\node.log.

See Section 11 above for copy-paste blocks.

I see "Previous shutdown was unclean - last block in progress: <X>". What do I do?

This is the shutdown tracker breadcrumb, not a withdrawal bug, and not a sign of corruption. It means the node was killed mid-block-commit (OOM, SIGKILL, power loss, container kill).

Do not run --cleanup. That wipes your chain back to genesis and forces a ~2.7-day resync. The advice floating in semi-official channels to use --cleanup for this message is wrong.

Run --rescue once, then restart normally:

docker exec blockdag-node bdag --rescue
docker restart blockdag-node
docker exec blockdag-node bdag attach --exec "eth.blockNumber"

If eth.blockNumber returns a number close to network head, you're done. Total recovery: ~30 seconds. No resync.

Stopgap if you can't upgrade yet (still on v1): remove the lockfile and restart.

docker exec blockdag-node rm data/<network>/lockfile
docker restart blockdag-node

That's a hack. Upgrade to v2.0.1 properly when you can.

When should I use --rescue vs --cleanup? (decision tree)

Almost always --rescue. The only legitimate use of --cleanup is when your data dir is genuinely corrupt and you've decided to start over.

Use --rescue when:

  • "Illegal withdrawal at block:X" on startup
  • "Previous shutdown was unclean" message
  • Node was OOM-killed, SIGKILL'd, power-lost, or container-killed mid-commit
  • "missing trie node" right after an unclean shutdown

Use --cleanup ONLY when:

  • Your disk had a hardware failure and the trie is genuinely partial
  • You manually edited or partially restored chaindata files
  • --rescue has been run and the node still won't boot for trie reasons
  • You've decided to start fresh and don't care about your existing chaindata

For all "I've decided to start fresh" cases, prefer the IPFS snapshot - faster than running --cleanup and resyncing from peers.

If unsure, run bdag --rescue --dry-run first - it reports what it would do without writing.

My node won't start with OOM / high memory. What knob do I tune?

PathDB is the hungry component. Two knobs: --state.diff-layers-max and --cache.

RAM--state.diff-layers-max--cache (MB)
8 GB1282048
16 GB (default)2564096
32 GB+5128192

Edit docker-compose.yml:

command:
  - --state.diff-layers-max=128
  - --cache=2048

Then docker compose up -d --force-recreate blockdag-node. Watch RSS for 30 minutes. If it still climbs past 80% of available RAM, drop --state.diff-layers-max one more step (128 -> 96).

If you're consistently OOM-killing on 8 GB, the honest answer is your box is undersized for steady operation - 16 GB is the comfortable floor.

How do I sync to network tip from scratch?

Two routes - snapshot (recommended) or peer sync.

Route 1 - IPFS snapshot (~30 min total): the installer does this automatically. See Section 12 above for the manual step-by-step.

Route 2 - sync from peers (~2.7 days): wipe your data dir and just start the node. It will discover peers and sync from genesis. Most operators don't choose this voluntarily.

Verify you're at tip: dashboard at http://127.0.0.1:8881 shows synced: true and your chain head matches any public RPC's head.

My node DB is corrupted. How do I resync from the DagTech IPFS snapshot?

Step by step:

  1. Stop the node: docker stop blockdag-node
  2. Move the broken data dir aside (don't delete until snapshot import succeeds): 
    sudo mv /var/lib/blockdag /var/lib/blockdag.broken.$(date +%s)
sudo mkdir -p /var/lib/blockdag
sudo chown -R bdag:bdag /var/lib/blockdag
  3. Resolve the latest snapshot CID (auto-updating IPNS pointer): 
    SNAPSHOT_CID=$(curl -s https://miner.dagtech.network/v201/snapshot-cid.txt)
echo "$SNAPSHOT_CID"
  4. Pull the snapshot via subdomain gateway (path-style gateways 404 for our snapshots): 
    curl -L "https://${SNAPSHOT_CID}.ipfs.dweb.link/" -o snapshot.tar.zst
curl -L "https://${SNAPSHOT_CID}.ipfs.dweb.link/sha256" -o snapshot.tar.zst.sha256
  5. Verify SHA before extracting - non-negotiable: 
    sha256sum -c snapshot.tar.zst.sha256
# expect: snapshot.tar.zst: OK
  6. Extract into the data dir: 
    sudo tar --use-compress-program=unzstd -xf snapshot.tar.zst -C /var/lib/blockdag/
sudo chown -R bdag:bdag /var/lib/blockdag
  7. Start the node: docker start blockdag-node
  8. Watch convergence at http://127.0.0.1:8881. Expect tip in 5-15 minutes with peers above 4.
  9. Once you're at tip and stable for an hour, you can delete the old broken dir: sudo rm -rf /var/lib/blockdag.broken.*

Snapshot lineage and hash history live at miner.dagtech.network/v201/snapshot-status.

Use the upgrade script - same pattern as v1 -> v2.0.1:

What if my peer count drops below 5?

First check is your inbound port:

curl -s "https://portchecker.io/api/query?host=YOUR_PUBLIC_IP&port=8130"

If port 8130 is closed: fix your NAT / firewall. That's 99% of low-peer cases. Behind hard NAT with no port mapping, you'll be peer-leeching and your count will sit low forever - the network won't dial you back.

If port 8130 is open and peers still stuck low for 15+ minutes:

  • Restart the node - bootstrap peers cycle
  • Check docker logs blockdag-node 2>&1 | tail -200 for "no route to host" or "dial timeout" patterns
  • Check your upstream isn't being filtered (ISP, hosting-provider AUP)
  • File a GitHub issue at github.com/Dagtechltd/dagtech-miner/issues with your public IP and the last 200 log lines
My miner gets "Expired job" - what's wrong?

"Expired job" from your miner means the pool sent a new mining.notify with clean_jobs=true before your miner finished the previous job. The miner discarded in-flight nonces and switched.

This is normal at low rates (a few per hour during DAG reorgs). It becomes a problem when:

  • You're seeing dozens per minute - the pool is sending clean_jobs=true on every notify (broken / mis-configured pool router), or your node is reorging constantly. Check pool config first, then node sync stability.
  • Bursts of expired jobs followed by zero hash output - vardiff death spiral. Your miner can't complete a job before the next clean_jobs arrives. Mitigation in v2.0.1 lands at the Sentry pool-side layer; on the node side, ensure your node isn't lagging the network.
  • One-off after a node restart - normal. The pool sees the chain advance and pushes a clean job. Ignore.

If you operate the pool too, check the pool config docs - clean_jobs should only fire on genuine GHOSTDAG reorgs that invalidate work.

Does this build phone home?

No.

The binary makes zero outbound calls to DagTech-controlled endpoints during normal operation. Specifically:

  • No telemetry, no usage analytics, no crash reporting service
  • No remote health check pings to dagtech.network
  • No stats upload, no anonymous metrics, no opt-out flag because there's nothing to opt out of
  • The local dashboard at 127.0.0.1:8881 is loopback-only; it doesn't post anywhere

The only times the binary touches DagTech infrastructure are operator-initiated:

  • install.sh / upgrade.sh fetch artifacts from miner.dagtech.network (that's the install path you chose)
  • Snapshot pull from IPFS (also install path)
  • If you set the optional update-check flag, the binary will check our releases manifest once a day - off by default

Want to verify? tcpdump -i any -n host not YOUR_PEERS while the node is steady-state. Outbound destinations should be peer IPs only.

CONSENSUS
Conflux-style deferred state commit (5-block window). Eliminates the "Zero state root" class of errors entirely by deferring state commitment until 5 blocks after the block lands. We've done the design work; implementation is in progress.
PLATFORM
Darwin binaries. v2.0.1 ships Linux + Windows only. macOS arm64 and amd64 binaries land in v2.1.0, ad-hoc codesigned at minimum.
SECURITY
GPG signature on SHA256SUMS. v2.0.1 is SHA-only over TLS. v2.1.0 adds Boss-signed GPG signature on SHA256SUMS.asc so the chain of trust extends past TLS termination at our CDN.
DASHBOARD
Wallet rate inflow vs outflow split panel. Today the dashboard shows combined wallet movement. v2.1.0 splits inbound rewards from outbound transactions so you can see your real revenue rate at a glance.
WATCHDOG
Sync watchdog will also disconnect the stale peer. Today the 30-second watchdog unsticks the import pipeline. v2.1.0 will additionally identify and drop the peer that fed the stale data, so the next reconnect picks a healthier one.
OBSERVABILITY
Structured event log over Unix socket. Optional newline-delimited JSON event stream on a local socket for operators running their own monitoring. Still local-only.
